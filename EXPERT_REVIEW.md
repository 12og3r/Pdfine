## 二、五位专家一致认可的设计决策

以下决策获得全员"正确"评价，无需修改：

1. **Document Model 作为核心枢纽** — 所有模块围绕统一的数据模型工作，是编辑器类产品的标准架构
2. **DocumentModel 不放入 React state** — 高频更新的编辑数据与 React 渲染周期解耦，性能考量正确
3. **弃用 fabric.js 做文本渲染** — fabric.js 无法做到字符级精确定位，自研 Canvas 渲染是唯一选择
4. **隐藏 textarea 捕获输入** — Google Docs 同款方案，处理 IME 的行业标准做法
5. **模块接口契约优先** — 接口先行、面向接口编程，适合多人并行开发

## 三、专家一致指出的关键问题（按严重程度排序）

### 🔴 P0 — 必须在开发启动前解决

#### 问题 1: Export Module 的"内容流清除"方案不可行

**共识**: PDF 规范专家 + 字体专家 + 架构师均指出此问题

PRD 提到"清除原始内容流中的特定区域再重新写入"，但 PDF 内容流是线性指令序列，无法简单"删除"某段文本的绘制指令（因为坐标状态、字体状态有累积依赖）。pdf-lib 也不具备解析和修改现有内容流的能力。

**专家建议的务实方案**:
- **v1.0 采用"白色矩形覆盖 + 重新绘制"**（简单、可靠、工期可控）
- 在 ExportModule 中预留 `IContentStreamEditor` 接口，后续迭代升级为精确内容流编辑
- PRD 应诚实地描述这一取舍，并说明对文件体积和可复制性的影响

#### 问题 2: 文本块溢出（Overflow）没有处理策略

**共识**: 排版引擎专家 + 编辑器专家 + 架构师均指出此问题

当用户修改文本导致内容超出原始 bounding box 时，PRD 完全没有讨论。这是用户编辑时**必然发生**的场景。

**专家建议的四层方案**:
1. **容错空间** — 允许 10-15% 的溢出容差
2. **自动收缩** — 按优先级尝试：缩小行距 → 缩小字间距 → 缩小字号
3. **UI 反馈** — 溢出时显示警告，属性面板提供"恢复原始排版"选项
4. **导出验证** — 导出前检查所有文本块是否溢出，溢出时弹窗确认

#### 问题 3: 测试策略完全缺失

**共识**: 全员一致

对于核心功能高度复杂的编辑器项目，没有测试策略等于没有质量保障。

**专家建议的最小测试体系**:
- **单元测试** — Parser（文本聚合）、LayoutEngine（断行算法）、ExportModule（坐标转换）各 100+ case
- **集成测试** — Parser → Layout → Render 全链路、Edit → Layout → Render 响应链路
- **往返测试（Roundtrip）** — PDF 导入 → 编辑 → 导出 → 重新导入，验证内容未丢失
- **性能测试** — 单次编辑 < 16ms、文本块重排 < 5ms 的基准验证
- **测试 PDF 样本库** — 至少 5-10 个不同复杂度的真实 PDF（英文、中文、多栏、加密、大文件）

### 🟡 P1 — 需要补充到 PRD 中

#### 问题 4: 文本块聚合算法复杂度被严重低估

**提出者**: PDF 规范专家、排版引擎专家

PDF 中的文本不是按"段落"组织的，而是离散的绘制指令。PRD 用一句话带过了聚合策略，但实际面对的边界情况极多：多栏布局、上标/下标、列表符号、RTL 混排、文本框嵌套等。

**补充建议**:
- 开发前用 5-10 个复杂 PDF 做 PoC 验证聚合算法
- 聚合阈值（x/y 距离、字高比例）需要可配置
- 允许用户手动调整/拆分/合并文本块（作为降级方案）

#### 问题 5: 字体处理的完整性不足

**提出者**: 字体排版专家、PDF 规范专家

| 问题 | 说明 | 建议 |
|-----|------|------|
| Type1/CIDFont 不支持 | opentype.js 无法解析这些格式 | v1.0 明确声明不支持，加载时检测并提示 |
| 字体子集缺字的回退策略太简单 | PRD 只说"回退到最接近的系统字体" | 实现分级策略：完整版本 → 同家族 → 同类型 → 通用备选 |
| Canvas 文本渲染精度 | subpixel rendering、hinting 导致 ±1-2px 偏差 | 坐标对齐到整数倍 1/dpr；PRD 补充精度声明 |
| 导出时字体嵌入 | pdf-lib 的子集化能力有限 | 考虑集成专业字体子集工具（如 fonttools/harfbuzz-subset） |

#### 问题 6: 坐标系统转换需要独立模块

**提出者**: 全员一致

Canvas（左上原点、Y 向下）vs PDF（左下原点、Y 向上）的转换贯穿所有模块。PRD 提到了这个问题但没有设计解决方案。

**补充建议**:
- 新增 `CoordinateTransformer` 模块，作为跨模块的基础设施
- 排版引擎内部统一使用"排版坐标系"（左上原点、Y 向下、单位 point）
- 仅在渲染和导出时做坐标转换
- 编写完整的单元测试（包括极端缩放、旋转的 case）

#### 问题 7: 断行算法选择和性能

**提出者**: 排版引擎专家

**建议的混合方案**:
- 实时编辑时：使用贪心断行（保证 < 5ms）
- 导出时：使用 Knuth-Plass 算法（保证最优排版）
- 字体度量缓存是性能关键：`fontId:charCode:fontSize → width` 的 Map 可将查询耗时降低 60-70%
- 增量重排（只重排受影响的行）是保证 16ms 帧率的必要条件

#### 问题 8: IME 输入法兼容性

**提出者**: 富文本编辑器专家

隐藏 textarea 方案正确，但实现中有多个深坑：
- composition 期间需要冻结光标、渲染"预编辑文本"的特殊样式
- textarea.value 与 Canvas 光标位置的同步（大文本块时有性能问题）
- Android Chrome 的部分输入法不触发 composition 事件
- Firefox 的 `beforeinput` 触发时机与 Chrome 不同

**建议**: 提前在 Week 1 进行 IME 兼容性预研

### 🟢 P2 — 后续迭代优化

#### 问题 9: 接口契约需要补充

**提出者**: 架构师

当前接口定义缺少：`IEditorCore` 顶层接口、事件总线的事件类型和参数规范、`IRenderEngine.setDirtyRect()`、`IFontManager.hasGlyph()` / `getFallbackFont()`、`IEditEngine.applyStyle()`

#### 问题 10: 事件总线粒度太粗

**提出者**: 架构师

`'modelChanged'` 无法告知 React 具体哪个 TextBlock 变了，会导致不必要的重渲染。建议细化为 `textChanged({pageIdx, blockId})` / `selectionChanged({blockId, range})` 等。

#### 问题 11: fabric.js 用于叠加层的集成复杂性

**提出者**: 架构师、编辑器专家

两个 Canvas 层（原生 + fabric）叠加时的事件分发很复杂。建议：要么统一用原生 Canvas（叠加层自己实现拖拽缩放，代码多但可控），要么把 fabric.js 的使用限制在极小范围内。

## 四、修正后的工期估算

### 全员共识：6 周 5 人不够，现实需要 8-10 周

| 模块 | PRD 原估计 | 专家修正 | 修正原因 |
|-----|-----------|---------|---------|
| PDF Parser | 3 周 (1人) | 4-5 周 | 文本块聚合算法需要反复调试 |
| Font Manager | 2 周 (1人) | 3-4 周 | 字体格式兼容、回退策略复杂 |
| Layout Engine | 3 周 (1人) | 5-6 周 | CJK 断行、两端对齐、性能优化 |
| Render Engine | 3 周 (1人) | 5-7 周 | HitMap 性能、选区渲染、脏区域 |
| Edit Engine | 3 周 (1人) | 5-7 周 | IME 兼容性、光标同步调试困难 |
| Export Module | 3 周 (1人) | 4-5 周 | 坐标转换、字体嵌入、覆盖方案 |
| 集成联调 | 1 周 (全员) | 2 周 | Canvas-React 集成、模块接口对齐 |
| 测试 | 1 周 | 2 周 | 从零建立测试体系 |
| **总计** | **6 周** | **8-10 周** | |

### 如果必须 6 周交付的压缩方案

**砍掉的功能**（从 P0 降级）:
- 加密 PDF 解锁 → 延后
- 图片编辑/插入 → 延后
- 注释高亮 → 延后
- 深色主题 → 延后
- CJK 标点禁则 → 简化
- 页面缩略图 → 延后

**接受的质量债**:
- 测试覆盖率 30%（后续补到 70%）
- 仅支持 Chrome（Safari/Firefox 有已知问题）
- 性能优化延后（大文件可能卡）
- 导出文件体积偏大（白色矩形覆盖方案）
